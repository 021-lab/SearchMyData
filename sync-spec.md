# ТЗ: Синхронизация списка клиент ↔ сервер

**Версия:** 1.0  
**Область:** логика синхронизации, протокол взаимодействия, контракт данных  
**Стек:** не регламентируется

---

## 1. Модель данных

### 1.1 Item — элемент списка

```json
{
  "id":        "string | number",  // уникальный стабильный идентификатор
  "line1":     "string",           // основной текст, обязательно, макс. 60 символов
  "line2":     "string | null",    // подпись, необязательно, макс. 80 символов
  "level":     "number",           // уровень вложенности: 0 = корень, 1+ = дочерний
  "collapsed": "boolean | null"    // свёрнута ли группа (только для родителей)
}
```

Поля `level` и `collapsed` могут отсутствовать — интерпретируются как `0` и `false`.

### 1.2 State — полное состояние списка

```json
{
  "seq":   42,
  "items": [ /* Item[] — упорядоченный массив */ ]
}
```

`seq` — монотонно возрастающий счётчик: увеличивается на 1 при каждом применённом патче.  
Порядок элементов в массиве определяет порядок отображения в списке.

### 1.3 Patch — патч

```json
{
  "seq":       43,
  "client_id": "string",
  "op":        "string",
  /* поля операции */
}
```

`seq` — порядковый номер, назначается клиентом, монотонно возрастает.  
`client_id` — идентификатор сессии клиента, генерируется при загрузке страницы.

## 2. Операции (op)

Контракт расширяемый: сервер и клиент обязаны игнорировать неизвестные `op`.

### 2.1 `item_add` — добавить элемент в конец списка

```json
{
  "op":    "item_add",
  "id":    "string",
  "line1": "string",
  "line2": "string | null"
}
```

### 2.2 `item_nest_add` — добавить вложенный элемент

Вставляется после последнего дочернего элемента родителя.

```json
{
  "op":        "item_nest_add",
  "id":        "string",
  "parent_id": "string",
  "line1":     "string",
  "line2":     "string | null"
}
```

### 2.3 `item_edit` — изменить текст элемента

```json
{
  "op":    "item_edit",
  "id":    "string",
  "line1": "string",
  "line2": "string | null"
}
```

### 2.4 `item_delete` — удалить элемент

Дочерние элементы удаляются вместе с родителем.

```json
{
  "op": "item_delete",
  "id": "string"
}
```

### 2.5 `item_move` — переместить элемент (и его группу)

```json
{
  "op":       "item_move",
  "id":       "string",
  "after_id": "string | null",  // null = переместить в начало списка
  "level":    "number"          // новый уровень вложенности элемента
}
```

Дочерние элементы перемещаются вместе с родителем; их уровни корректируются на `delta = new_level - old_level`.

### 2.6 `item_collapse` — свернуть / развернуть группу

```json
{
  "op":        "item_collapse",
  "id":        "string",
  "collapsed": "boolean"
}
```

## 3. Маппинг действий пользователя → операции

| Действие пользователя | Операция |
|---|---|
| Нажать `+`, заполнить форму | `item_add` |
| Свайп вправо → Вложенный, заполнить форму | `item_nest_add` |
| Свайп вправо → Изменить, сохранить | `item_edit` |
| Свайп влево (прямо) | `item_delete` |
| Свайп влево + вверх | `item_move` (after_id = id элемента двумя позициями выше) |
| Свайп влево + вниз | `item_move` (after_id = id следующего элемента) |
| Долгий тап + перетаскивание | `item_move` (after_id и level вычисляются по финальной позиции) |
| Тап по элементу с дочерними | `item_collapse` |
| Кнопка Undo | обратная операция к последнему патчу в `pending` (см. п. 5.4) |

## 4. Локальное хранилище клиента (localStorage)

```json
{
  "last_known_state": {
    "seq":   42,
    "items": []
  },
  "pending_patches": [
    { "seq": 43, "client_id": "...", "op": "item_add", ... },
    { "seq": 44, "client_id": "...", "op": "item_edit", ... }
  ],
  "last_seq": 44
}
```

| Ключ | Описание |
|---|---|
| `last_known_state` | Последнее подтверждённое сервером состояние |
| `pending_patches` | Патчи, применённые локально, но ещё не подтверждённые сервером |
| `last_seq` | Счётчик для генерации seq следующего патча |

## 5. Логика клиента

### 5.1 Инициализация (каждый раз при загрузке страницы)

```
1. Читаем last_known_state и pending_patches из localStorage
2. Рендерим DOM из last_known_state
3. Применяем pending_patches к DOM (оптимистично, визуально)
4. Страница готова к работе — до получения ответа сервера
5. [фон] GET /state
6. Фильтруем pending: оставляем только seq > response.seq
7. Если pending не пусто: POST /patches
8. Обновляем last_known_state ← response.state (после ACK)
9. Удаляем подтверждённые патчи из pending
```

Если `last_known_state` отсутствует (первый запуск): шаги 2–3 пропускаются,  
DOM строится целиком из ответа сервера на шаге 5.

### 5.2 Нормальная работа (действие пользователя)

```
1. Записываем патч в localStorage (pending_patches) ← write-ahead
2. Применяем патч к DOM
3. Если нет активной отправки: flush pending → POST /patches
4. При ACK { seq: N }:
     - удаляем из pending все патчи с seq ≤ N
     - обновляем last_known_state
```

**Write-ahead обязателен:** localStorage пишется до изменения DOM,  
чтобы при краше между шагами 1 и 2 патч не потерялся.

### 5.3 Потеря соединения

```
- Патчи накапливаются в pending (localStorage)
- Повторные попытки отправки с backoff: 2s → 4s → 8s → 16s → 30s
- При восстановлении: стандартный flush pending
```

### 5.4 Undo

Undo работает только для патчей, ещё находящихся в `pending` (не подтверждённых).  
Клиент генерирует обратный патч и добавляет его в `pending`:

| Исходный op | Обратный op |
|---|---|
| `item_add` | `item_delete` |
| `item_nest_add` | `item_delete` |
| `item_edit` | `item_edit` (с предыдущими значениями) |
| `item_delete` | `item_add` или `item_nest_add` |
| `item_move` | `item_move` (с предыдущими after_id и level) |
| `item_collapse` | `item_collapse` (с противоположным collapsed) |

Undo подтверждённого патча: только через новый прямой патч (нет возможности отозвать ACK).

## 6. API бэкенда

### 6.1 `GET /state`

Возвращает текущее состояние.

**Response 200:**
```json
{
  "seq":   42,
  "items": [ /* Item[] */ ]
}
```

### 6.2 `POST /patches`

Принимает массив патчей, применяет их, возвращает новое состояние.

**Request:**
```json
{
  "patches": [
    { "seq": 43, "client_id": "abc", "op": "item_add", ... },
    { "seq": 44, "client_id": "abc", "op": "item_edit", ... }
  ]
}
```

**Response 200 (успех):**
```json
{
  "ok":  true,
  "seq": 44,
  "state": { /* полный State после применения */ }
}
```

**Response 409 (конфликт версий):**
```json
{
  "ok":    false,
  "error": "seq_conflict",
  "state": { /* актуальное State сервера */ }
}
```

При 409: клиент перезагружает `last_known_state` из ответа, переприменяет pending к нему и повторяет отправку.

## 7. Логика бэкенда

### 7.1 Применение патчей

1. Патчи применяются строго по порядку `seq`
2. Патч с `seq ≤ state.seq` — идемпотентно пропускается (уже применён)
3. Патч с `seq > state.seq + 1` — разрыв последовательности → 409
4. После каждого успешно применённого патча: `state.seq++`
5. Состояние персистируется **до** отправки ACK

### 7.2 Правила применения операций

| Op | Правило |
|---|---|
| `item_add` | Добавить в конец `items` |
| `item_nest_add` | Найти `parent_id`, вставить после его последнего дочернего |
| `item_edit` | Найти `id`, обновить `line1`, `line2` |
| `item_delete` | Найти `id`, удалить его и все элементы с `level > parent.level` непосредственно следующие за ним |
| `item_move` | Извлечь элемент и его группу, вставить после `after_id`, применить delta к уровням группы |
| `item_collapse` | Найти `id`, установить `collapsed` |

Если элемент с указанным `id` не найден — патч пропускается без ошибки.

## 8. Расширяемость контракта

Для добавления нового типа действия:
1. Ввести новый `op` (строка в snake_case)
2. Описать его поля в контракте
3. Старые клиенты и сервер игнорируют неизвестный `op`

Для добавления нового поля в `Item`:
1. Описать поле как необязательное
2. Отсутствие поля = значение по умолчанию

Версионирование контракта через заголовок `X-Contract-Version` (опционально).

## 9. Гарантии и ограничения

| Гарантия | Комментарий |
|---|---|
| Один клиент одновременно | Несколько вкладок не поддерживаются |
| Потеря соединения без потери данных | При условии write-ahead в localStorage |
| Идемпотентность повторной отправки | Гарантируется seq-фильтрацией на сервере |
| Порядок патчей сохранён | Клиент отправляет батчем строго по seq |

| Известное ограничение | Комментарий |
|---|---|
| Undo подтверждённых патчей | Невозможен без специального rollback API |
| Другой браузер / устройство | Pending из localStorage недоступны |
| Очистка localStorage | Pending патчи теряются |
